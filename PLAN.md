## Context: why M16

M15 read the app-auth policy rules and derived a strength BAND per policy, and it wired that band onto the `risk` surface as an **evidence column**. But it deliberately did NOT change how the risk SCORE is computed — that was stated three times in the M15 plan as out-of-scope, and closed as a pinned characterization test (`expected-red.test.ts`, "M15 Phase E — deferred limitation") kept green so M15 shipped at 0 expected-fail.

The result is a **self-contradiction on a shipped surface**: `rankRisk` still weights the gate by M8's org-default-vs-custom PRIOR (`DEFAULT_GATE_MULT = 2` — org-default scores 2×, "riskier"), while the band column right beside it shows the evidence that *inverts* that prior. On the live capture, GitHub (org default, **two-factor** band) outscores Confluence (custom Strict-Auth, **single-factor** band) at equal reach — the score says GitHub is riskier; its own band says the opposite. The `risk` render even prints "trust the band (band-aware scoring is future work)" to paper over the gap.

M16 closes it: derive the gate multiplier from the captured band, so the score agrees with the evidence the tool already displays. This is the one debt M15 named for the very next milestone, and it is on the project's honesty arc (M12→M15): don't ship a number that contradicts your own evidence.

## Milestone 16: band-aware risk scoring

**Goal:** the `risk` gate multiplier for an **App** row is derived from its gate's captured strength BAND (weaker band ⇒ higher risk) wherever rules were captured; `unknown` bands score neutral; the score no longer contradicts the band column. The M15 characterization pin flips to a green assertion of the corrected ordering, and the "band-aware scoring is future work" caveat is retired. No surface claims a scoring basis it doesn't have.

**Rails (unchanged):** `src/analysis` (`rank-risk.ts`) + `src/core` stay pure — no I/O, no new capture (M16 reads only the `AppAuthPolicyRule` records M15 already captures + the `strengthResolver` M15 already built). No envelope bump (rules already ride as `GraphEnvelope.policyRules`, M15 Phase D). Ranking stays O(N+E) — the resolver is an O(1) map read per row, no per-subject traversal. The M8 fixture oracle is a locked regression guard: `rankRisk` called WITHOUT a resolver must produce byte-identical scores to today (see Phase A).

### Phase 0 — design lock (no code; the multiplier table is the deliverable)

Two scope decisions and one constant table, recorded here before code (the M15 Phase-0 discipline). Both decisions are **revisitable** — flagged so a reversal is a localized change.

- **S1 — band scoring is APP-only; GROUP rows keep the session prior.** `rankRisk` scores both App and Group rows. Only App gates have a captured band (`AppAuthPolicyRule`). Session-policy rule strength is **M15 D2 deferred** (`okta_policy_rule_signon` never captured), so there is no band for a Group's session gate. M16 therefore changes App-row gate weighting only; **Group rows keep M8's session prior unchanged** (`session-policy` ⇒ 1×, `none` ⇒ 2×). *If session-rule bands ever land (the D2 fast-follow):* Group scoring gets the same treatment additively; nothing in the App path changes.

- **S2 — `unknown` band scores NEUTRAL (per the roadmap), retiring the org-default _scoring_ prior on paths without captured org-default rules.** When a resolver is present but an App's gate bands `unknown` — the documented tfstate divergence (the org-default system policy's rules are never in state) or any custom policy with no captured rules — its gate multiplier is **neutral (1× baseline)**, NOT the M8 org-default 2× prior. Consequence, stated plainly: on the **tfstate** path the org-default gate stops scoring as "riskier by default" (we no longer penalize org-default-ness itself when we can't read its rules); on the **live** path the org-default resolves to the system policy's real band and scores from it. The `gate`/`gatePrior` FACTUAL label and column are unchanged — only the score weight moves. *Alternative (rejected, revisitable): prior-as-fallback* — keep M8's 2× when the band is `unknown`. Rejected because the one case we can check (the kicker) showed the prior is unreliable; keeping it as the blind-path fallback trusts a heuristic we've watched fail. But it's the more conservative migration if a tfstate ranking shift proves surprising — reversing = one branch in `appGateMult`.

**Constant table — band ⇒ gate multiplier** (recommended defaults; the ORDERING is the locked invariant, the exact constants are pinned to the oracle + revisitable, à la M15's fact table). Integer scale (×2 of the old prior) so scores stay whole and the weak/neutral endpoints are scale-continuous with M8:

| Band | Ordinal | Gate mult | Rationale |
|---|---|---|---|
| `single-factor` | 1 | **4** | weakest documented way in ⇒ highest gate risk (= 2× the old default-gate weight; the kicker's weak side) |
| `two-factor` | 2 | **2** | the common 2FA baseline ⇒ neutral risk |
| `phishing-resistant-2fa` | 3 | **1** | strongest realistic gate ⇒ dampens exposure below baseline |
| `deny-all` | 4 | **0** | nobody gets in ⇒ zero access risk from the gate (sorts the app to the floor) |
| `unknown` | — | **2** | NEUTRAL (S2) — no evidence, sits at the 2FA baseline; never amplifies or dampens |

**Locked invariant (the property test, not the constants):** `mult(single-factor) > mult(two-factor) > mult(phishing-resistant-2fa) > mult(deny-all)`, `mult(unknown) == mult(two-factor)` (neutral), `mult(deny-all)` is the minimum. *Sub-decision, revisitable:* `deny-all = 0` (hard floor — a deny-all app carries no gate exposure) vs a small positive (keeps it non-collapsing if a deny-all gate ever needs to rank above a zero-reach row); `0` chosen — reach×0 sorting an all-DENY app last is the honest reading.

**Phase 0 status: COMPLETE (2026-07-21).** S1 (App-only band scoring; groups keep the session prior) + S2 (`unknown` → neutral, retiring the org-default scoring prior where org-default rules aren't captured) recorded above; constant table locked as `BAND_GATE_MULT` (`single-factor`4 / `two-factor`2 / `phishing-resistant-2fa`1 / `deny-all`0 / `unknown`2). Ordering is the tested invariant; constants revisitable.

### Phase A — the band-aware score (pure analysis + tests)

## Roadmap (rewrite this file per milestone as each starts)

**M17+ candidates — practitioner adoption (analysis 2026-07-18; pick by value-per-lift when M16 ships):**

- **CI drift gate** (small lift, highest leverage): `coverage --check [--min-coverage N]` → nonzero exit on gaps/stale + a markdown summary artifact. Turns the tool from a one-off inspection into a scheduled control (nightly job catches click-ops drift the morning after). Most of the value of "reconciliation as a practice" for ~a day of work. *(The leading M17 candidate.)*
- **OAuth service-app auth** (medium lift, removes the biggest enterprise blocker): SSWS tokens are org-wide and inherit their creator; real orgs mandate scoped OAuth (`okta.groups.read` etc., DPoP). The read-only story becomes enforceable by scopes, not by trusting a Read-Only-Admin token. CLAUDE.md already anticipates this switch.
- **Access-review evidence export** (differentiated — nothing else does this): dated, self-contained HTML/CSV report of "who can reach what, via which group/rule/policy, and its IaC status" — direct evidence for SOC2/ISO access-certification. Builds almost entirely on existing trace + coverage.
- **Multi-state merge** (real-org blocker): accept `--state` repeated / a directory; real teams split Terraform across workspaces, so one tenant ↔ many state files. Presence-union is semantically simple; the work is CLI + dedupe rules.
- **Scope honesty at the product level**: the report says "coverage" but classifies 6 kinds; real orgs also manage password policies, MFA enrollment, network zones, admin roles, IdPs in Terraform. Either widen kinds incrementally or print a "measured kinds: …" scope line on every report (cheap, honest, M14's lesson generalized).
- **Session-policy rule strength (D2 un-defer)**: M15/M16 scope app-auth rules only; global session-policy rules (`okta_policy_rule_signon`) stay a prior on the risk `session-policy` label + trace "session gate". A second capture + parser variant + a session-band set (`mfa_required`/`session_lifetime`/`primary_factor`) would give Group rows the same grounded treatment M16 gives App rows. Additive; nothing in the app-auth path changes.
- **Distribution**: publishable package (`npx <tool> coverage ...`), 5-minute README quickstart, and the rename off the `okta-` prefix (CLAUDE.md trademark note) — adoption dies at `git clone` + `tsx`.
